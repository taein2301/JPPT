# 다국어(타 언어) 전환 프로젝트 상세 설계서

## 1. 문서 목적

이 문서는 기존 JPPT(파이썬 기반 템플릿)를 **다른 언어 기반 프로젝트로 재구축**할 때 필요한 기준 문서다.
목표는 다음과 같다.

- 기능 누락 없이 재구축한다.
- 팀원마다 다른 해석이 나오지 않도록 요구사항을 명확히 한다.
- 구현 우선순위, 워크플로우, 산출물, 검증 기준을 표준화한다.
- 이후 언어가 바뀌어도 재사용 가능한 아키텍처/운영 원칙을 만든다.

---

## 2. 프로젝트 전환 범위

### 2.1 포함 범위 (In-Scope)

1. CLI 기반 실행 구조 재구축
   - daemon(start) 모드
   - batch(one-shot) 모드
2. 환경별 설정 관리 구조 재구축
   - dev/prod 분리
   - 예시 파일(.example) + 실제 파일(ignored) 정책
3. 로깅/예외/재시도/신호처리/HTTP/텔레그램 알림 유틸리티 재구축
4. 테스트/정적검사/포맷터/CI 파이프라인 동등 수준 확보
5. 프로젝트 생성 스크립트(create_app) 동등 기능 또는 개선 버전 제공
6. 문서 체계 재정비
   - PRD
   - 아키텍처 문서
   - 운영 가이드
   - 온보딩 체크리스트

### 2.2 제외 범위 (Out-of-Scope)

1. 특정 도메인 비즈니스 로직(사용자 맞춤 core 구현)
2. UI/웹 프론트엔드 추가 개발
3. 기존 파이썬 코드와의 영구 양방향 동기화 도구 개발

---

## 3. 전환 원칙 (핵심 의사결정 기준)

1. **행동 동일성 우선**: 언어가 달라도 사용자 관점 동작은 동일해야 한다.
2. **관찰 가능성 보존**: 로그/에러코드/알림 체계는 오히려 강화한다.
3. **운영 안전성 우선**: graceful shutdown, retry, timeout, idempotency 보장.
4. **테스트 선행**: 기존 기능을 스펙화(테스트/시나리오)한 뒤 이식한다.
5. **문서-코드 동기화**: 변경 시 문서를 함께 수정하는 프로세스를 강제한다.

## 4. 기능 요구사항 상세 리스트 (Feature Inventory)

아래 항목은 신규 언어 프로젝트에서 반드시 제공해야 하는 기준 기능이다.

## 4.1 CLI

### 필수 명령

- `start`: 상주 실행
- `batch`: 1회 실행
- `--version`: 버전 출력

### 공통 옵션

- `--env`: 환경 선택 (기본 `dev`)
- `--config`: 설정 파일 경로 오버라이드
- `--log-level`: 로그 레벨 오버라이드
- `--verbose`: 상세 출력

### 검증 기준

- 잘못된 옵션 입력 시 비-0 종료코드
- 도움말 출력 일관성(명령/옵션 설명 포함)
- CI 환경에서도 비대화형 실행 가능

## 4.2 설정 시스템

### 구조

- `config/dev.yaml.example`, `config/prod.yaml.example` 유지
- 실제 실행 파일 `config/dev.yaml`, `config/prod.yaml`는 git ignore
- 환경변수 오버라이드 지원

### 요구사항

- 스키마 기반 검증(필수/선택, 타입, 범위)
- 시작 시 설정 유효성 실패하면 즉시 종료
- 민감정보 마스킹 출력

## 4.3 로깅 시스템

### 필수 기능

- 콘솔 + 파일 동시 출력
- 일 단위 로테이션
- 보존 기간 관리
- 실행 모드(start/batch)에 따른 로그 파일 구분
- 로그 상관관계 식별자(correlation id) 확장 포인트

### 로그 필드 표준

- timestamp
- level
- module/function/line
- message
- env
- app_name
- run_mode

## 4.4 예외/오류 체계

### 커스텀 오류 계층

언어별 형태는 달라도 아래 도메인 분류를 유지한다.

- ConfigurationError
- ValidationError
- ServiceError
  - HttpClientError
  - TelegramError
- RetryExhaustedError

### 요구사항

- 사용자 메시지(친화적)와 내부 로그 메시지(디버그용) 분리
- 실패 유형별 종료코드 정책 문서화
- 알림 전송 실패가 본 작업 실패로 전파되는 조건 명시

## 4.5 HTTP 클라이언트

### 필수 기능

- 공통 base URL
- 타임아웃
- 재시도 연동
- 상태코드 검증
- JSON 파싱 오류 처리
- 컨텍스트 생명주기 관리(초기화/종료 보장)

## 4.6 재시도(Retry)

### 기본 정책

- 최대 3회
- 지수 백오프(예: 1s, 2s, 4s)
- 재시도 대상 오류 화이트리스트

### 확장 정책

- 함수 단위 오버라이드
- 재시도 후 최종 실패 시 원인 체인 보존

## 4.7 텔레그램 알림

### 필수 기능

- enabled 플래그
- bot token/chat id 설정 검증
- 성공/실패 알림 템플릿
- 네트워크 실패 시 graceful degrade(로그로 대체)

## 4.8 신호 처리 / 종료

### 필수 시나리오

- SIGINT/SIGTERM 수신 시 작업 중단 플래그 set
- 현재 진행 중 작업의 안전한 종료
- 리소스(파일 핸들/네트워크/락) 해제

---

## 5. 비기능 요구사항 (NFR)

## 5.1 성능

- batch 모드 cold start 시간 목표 정의 (예: < 1.5s)
- 메모리 상한선 정의 (예: steady-state < 300MB)

## 5.2 안정성

- 24시간 daemon soak test 통과
- 외부 API 간헐 실패 상황에서 복구 가능

## 5.3 보안

- 민감정보 로그 금지
- 설정 파일/환경변수 취급 정책 문서화
- 의존성 취약점 점검 자동화

## 5.4 유지보수성

- 정적 분석 0 critical
- 타입/린트/테스트 파이프라인 필수
- 모듈 경계(core vs infra) 명확화

## 5.5 관측성

- 오류율, 재시도 횟수, 알림 성공률 등 핵심 메트릭 수집 포인트 정의

## 6. 신규 프로젝트 생성 워크플로우 (상세)

## 6.1 준비 단계

1. 대상 언어/런타임/패키지 매니저 확정
2. 빌드/배포 타깃(OS, 컨테이너 여부) 정의
3. 기존 JPPT 기능 매핑 테이블 작성(아래 7장 템플릿 사용)
4. 팀 코딩 규칙/브랜치 전략 확정

## 6.2 설계 단계

1. 아키텍처 문서 작성
   - 레이어 경계
   - 설정/로깅/에러 정책
2. CLI 명세 확정
3. 설정 스키마 정의
4. 예외/종료코드 매트릭스 정의
5. 테스트 전략(단위/통합/e2e) 문서화

## 6.3 부트스트랩 단계

1. 저장소 초기화
2. 디렉터리/모듈 골격 생성
3. 샘플 명령(`start`,`batch`) 최소 동작 구현
4. 기본 로거/설정 로더/예외 타입 추가
5. CI 기본 파이프라인 연결

## 6.4 기능 이식 단계 (권장 순서)

1. 설정 시스템
2. 로깅
3. 예외 + 종료코드
4. HTTP 클라이언트
5. 재시도
6. 텔레그램
7. 신호처리
8. create_app 스크립트

## 6.5 검증 단계

1. 기능 동등성 테스트
2. 장애 주입 테스트(타임아웃/5xx/네트워크 단절)
3. 성능/안정성 테스트
4. 문서-코드 일치 점검

## 6.6 릴리즈 단계

1. 버전 태깅 규칙 적용
2. 변경 로그 작성
3. 마이그레이션 가이드 배포
4. 운영 점검 체크리스트 수행

## 8. 테스트 전략 (상세)

## 8.1 테스트 레벨

1. 단위 테스트
   - 모듈별 정상/예외/경계값
2. 통합 테스트
   - 설정+로깅+실행기+외부 연동 조합
3. E2E 테스트
   - 실제 CLI 커맨드 호출 기반
4. 회귀 테스트
   - 기존 버그 재현 시나리오 고정

## 8.2 필수 시나리오 카탈로그

- 설정 파일 누락
- 설정 타입 오류
- API 타임아웃
- API 5xx + 재시도 후 성공
- API 5xx + 재시도 소진
- 텔레그램 비활성화/활성화
- start 모드에서 SIGTERM 수신
- batch 모드 성공/실패 종료코드

## 8.3 품질 게이트(예시)

- 테스트 커버리지 >= 80%
- 정적분석(린트/타입) 통과
- 보안 스캔 통과(고위험 취약점 0)

## 11. 수용 기준 (Definition of Done)

아래를 모두 만족해야 전환 완료로 본다.

1. 필수 기능(4장) 100% 구현
2. 필수 시나리오 테스트 통과
3. 문서 패키지(10장) 완성
4. 운영 핸드오버 세션 완료
5. 신규 팀원이 문서만 보고 로컬 실행 및 배포 리허설 가능

1. 언어 생태계 차이로 동일 라이브러리 부재
   - 대응: 기능 요구를 인터페이스로 분리하고 대체 라이브러리 평가표 작성
2. 운영 시그널/프로세스 모델 차이
   - 대응: OS/런타임 별 종료 동작 사전 검증
3. 테스트 자산 부족
   - 대응: 이식 전 현재 동작을 Golden Test로 고정
4. 문서 부실로 인수인계 실패
   - 대응: PR 템플릿에 문서 갱신 체크 강제

## 15. 부록: 초기 결정 로그 템플릿

아래 템플릿으로 의사결정을 남긴다.

```md
# ADR-0001 제목
- 날짜:
- 결정사항:
- 대안:
- 결정 이유:
- 영향도:
- 후속 작업:
```

이 로그를 남기면 언어 전환 후에도 "왜 이렇게 구현했는지"를 추적할 수 있다.
